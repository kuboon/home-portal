# home portal (ホムポタ)

AI エージェントとの協働を前提とした、Discord
ライクなコミュニティチャットプラットフォーム。

---

## 概要

- **正式名称**: home portal
- **愛称**: ホムポタ
- **コンセプト**: Discord 的なマルチテナント型チャットに、MCP
  によるエージェント拡張を標準装備する
- **想定利用者**: 家族や小規模グループで、会話に AI
  エージェントを参加させたいユーザ

---

## アーキテクチャ

### ホーム (Home)

- Discord の「サーバー」に相当する単位
- 複数のホームが存在し、ユーザは複数のホームに所属できる
- 各ホームには**管理者権限 (admin) のユーザが存在する**
  - **admin は複数人 (複数アカウント) 持てる**
- ホーム内のメンバー同士でチャットが可能
- **最大参加人数: 40 人** (学校の 1 クラス程度)
  - より大規模に使いたい場合は複数ホームを建てる方針

#### ホームの作成・削除

- **ホーム一覧画面**: 自分が所属する全ホームを一覧表示する画面を持つ
- この画面から**ホームの作成**ができる
  - 作成者が**初代 admin** になる
- この画面から**ホームの削除**ができる
  - 削除は **admin のみ**実行可能
  - (物理削除か論理削除かは [要決定](#未決事項--todo))

#### 管理者の任命・降格

- admin は他のメンバーを **admin に任命**、また admin を **member に降格**できる
- 管理者ユーザ死亡時などの継続運用は、**生前に admin
  権限のエージェントを登録しておく**ことで対処
  (プロダクト側では特別な仕組みを持たず、運用に委ねる)
- 推奨ガード: **最後の 1 人の admin は降格・除外できない** (zero-admin で
  request→approve が詰むのを防ぐ)

### ユーザ

- 1 アカウントが以下を複数持てる:
  - **passkey** (複数デバイスでの認証用)
  - **ホーム所属** (複数コミュニティ横断)
  - **web push サブスクリプション** (複数デバイスでの通知受信)

### 人間ユーザとエージェントユーザ

- エージェントは独立したユーザアカウントとして扱うが、**所有者となる人間ユーザ**に紐づく
- **チャット画面上の UI 区別はなし**
  (発言内容で自然に判断する想定、家族・小規模グループでは問題にならない)
- **プロフィール表示相当の画面**でのみ「誰のエージェントか」を表示
- **エージェント登録時に admin role を付与するか選択できる**
  (管理者のみが付与可能)
  - admin 付きエージェント: 管理操作を自ら実行でき、他者の request を approve
    もできる
  - admin なしエージェント: 一般メンバーと同様に request を発行し、admin (人間
    or admin エージェント) の approve を待つ

### ゲストユーザ

ホームのメンバーではないが、**招待された特定スレッドにのみアクセスできる**ユーザ。

- ホームの Membership は持たない (= メインチャンネルや他スレッドは見えない)
- 招待されたスレッドの**閲覧・post・リアクションが可能**
  (そのスレッド内では一般参加者とほぼ同等に振る舞える)
- post / リアクションすれば、そのスレッドの **participant (joined)**
  になり通知も受ける
- **スレッドが archived になった後も閲覧は可能** (post
  できないのは一般参加者と同じ)
- **40 人のメンバー上限にはカウントしない** (メンバーではないため)
- レート制限など他のルールは一般ユーザと同一

#### ゲストの招待

- **そのスレッドの参加者であれば誰でも** (ゲスト自身も含む) ゲストを招待できる
- 招待はスレッド単位 (ホームへの ephemeral 招待とは別系統)
- = ゲスト招待は管理操作ではなく、参加者の一般操作

> 🚧 **残る注意点**:
>
> - **repost のジャンプ漏洩**: ゲストが見るスレッド内の repost
>   を辿ると、原本がメインチャンネルや未招待スレッドにある場合に非公開部分が露出する。ゲストには
>   repost のジャンプを禁止し**プレビュー止まり**にする
> - **ゲストの post が pickup された場合**: 別スレッド (ゲスト未招待) に repost
>   されたとき、ゲストをそのスレッドの participant
>   にすると未招待スレッドへアクセスさせてしまう。pickup 起点の participant
>   化はゲストには適用しない、等の制御が要る
> - ゲストの認証も passkey (おそらく yes)

## 認証・オンボーディング

**passkey 一本化**。パスワードは持たない。

### 登録フロー (招待受諾時)

招待を受けた時点で手間が多いと離脱するため、**登録は passkey
登録のみで完了**とする。

1. 招待 (QR / URL) を踏む
2. **passkey 登録** → これで登録完了・ホーム参加

### 登録後に促す (ブロックしない)

- **add to home screen** (PWA としてインストール)
- **web push 通知の有効化**

上記は登録のゲートにはしないが、本プロダクトは push
前提の体験なので**登録直後に強く促す**。

> ⚠️ **設計上の論点**: 当初の「web push 必須化
> (通知が届かないユーザを構造的に排除)」と、この「passkey
> だけで参加完了」はトレードオフの関係。push をどこまで強制するかは要決定 (例:
> 完全任意 / 繰り返しプロモート / 「post
> する前に有効化が必要」等の段階的ゲート)。

## チャンネルとスレッド

Discord のチャンネル / スレッド概念を整理し、以下のシンプルな構造にする。

### メインチャンネル

- 各ホームに**1 つだけ**存在する概念上のチャンネル (Slack の `#general`
  のような位置づけ)
- ホームメンバー全員が参加
- **専用のテーブルは持たない**: メインチャンネルの post は「`thread_id`
  を持たない (= `null`) post」として表現される
- (スレッドの post は `thread_id` を持つ。両者の違いは `thread_id` の有無のみ)

### スレッド

- メインチャンネルでの会話の途中、あるいは**任意のタイミング**で作成できる
- スレッドは **uniq な `thread_id`** を持つ独立したエンティティで、**特定の単一
  post には紐づかない**
- スレッドは**新しいチャンネルのように振る舞う** (チャンネルの役割も兼ねる)
- **スレッドタイトルは作成者が編集可能** (admin は全権限として編集可)

#### pickup (随時可能)

- スレッド作成時だけでなく、**いつでも既存スレッドに対して post を pickup (=
  repost) できる**
- 生成時にだけ紐づける仕様だと、元チャンネル側で話題が続いたときに後追いできないため
- pickup された post は、そのスレッドに **repost** として現れる (詳細は
  [repost 節](#repost-汎用機能) 参照)

#### 参加者

- 参加者状態は **`ThreadParticipant`** テーブルで管理 (state: `joined` / `left`)
- **初期参加者 (participant)**: 以下の和集合が `joined` 状態で登録される
  - pickup された原本 post の**投稿者**たち (リアクション者は含まない)
  - **スレッドの作成者本人**
- **空スレッドの場合**: 初期参加者は作成者のみ
- **以降の参加者化トリガ** (新規 join もしくは left → joined への復帰):
  - そのスレッドへ post した
  - **そのスレッドに属する post (= `thread_id = T` を持つ post)
    にリアクションした**
  - repost の**リンク先 (原本)** へのリアクションは**含まない**
- **明示的な leave が可能** (mute は廃止し、leave に一本化)
  - leave した参加者が再度 post またはリアクションすると、**自動的に joined
    へ復帰**する
- **通知スコープ**: スレッドへの返信は、`joined` 状態の参加者のみに web push
  が飛ぶ

#### 発見性 (サイドパネル)

- ホーム内で発生した**すべてのスレッドがサイドパネルに表示される**
- サイドパネル内では **参加しているスレッド / 未参加のスレッド**
  をグループ化して表示
- 未参加スレッドも**中身を全文読める** (読むだけでは参加者化しない)

#### 双方向リンクによる誘導

- ある post が別スレッドで repost された場合、**原本 post から repost 先
  (子スレッド) へ辿れる導線**を表示する
- これにより、「スレッドに話題が枝分かれしていったこと」を元の場所にいるメンバーも認識できる
- 「メインチャンネル →
  スレッドに人が流れる」だけでなく、「スレッド先にも話が広がっている」ことが可視化される

#### 空スレッドの作成

- pickup なしで、**タイトルのみ指定して空スレッドを作成可能**
- 作成者が初期参加者として自動的に登録される (上述)

#### 入れ子スレッド

- UI 上は入れ子可能だが、内部実装としては**兄弟スレッド**として扱う
- スレッド内 post からの pickup でも、上記の pickup 挙動と同じ (新 `thread_id`
  が発行される)

#### スレッドの状態 (物理削除なし)

スレッドは**物理削除しない**。状態は 2 つ。

| 状態       | 説明                     | 振る舞い                                                                      |
| ---------- | ------------------------ | ----------------------------------------------------------------------------- |
| `active`   | 通常状態                 | post 可能。サイドパネルの参加/未参加グループに表示                            |
| `archived` | 非アクティブ化された話題 | **読み取り専用 (追加 post 不可)**。サイドパネルの「アーカイブ」グループに格納 |

##### 自動アーカイブと archived スレッドの振る舞い

- スレッドは **7 日間追加 post がなければ自動で `archived`** になる
- **un-archive 機能は提供しない** (一度アーカイブされたら戻せない)
- **archive 化の時点で、全 participant が leave 状態になる**
  (以降の通知は誰にも飛ばない)

###### archived スレッドで可能 / 不可能な操作

| 操作                                                 | 可否          |
| ---------------------------------------------------- | ------------- |
| 追加 post                                            | ❌ 不可       |
| post へのリアクション付与                            | ❌ 不可       |
| 配下 post の単発 repost (= 別 active スレッドで引用) | ✅ 可能       |
| 管理者による post 非表示                             | ✅ 可能       |
| 管理者によるスレッドタイトル編集                     | ✅ 可能       |
| 新着通知の発生                                       | ❌ 発生しない |

- archive 後に話題を続けたい場合は、archived スレッド内の post を **repost
  して別スレッドを作る**
  - このとき**新スレッドの participants は引き継がれず再構成される** (= 作成者 +
    pickup された原本 post の投稿者から再スタート)
  - 「過去の参加者全員を巻き込まずに、改めて話を始められる」クリーンな継続

##### スレッドの「非表示」操作 (= 配下 post の一括非表示)

- **thread 自体には hidden 属性を持たせない**
- スレッドへの「非表示」操作は、**そのスレッド配下の post をまとめて非表示
  (論理削除)** する操作として定義
- 論理削除は post レイヤーに一元化される (下記 post の可視性参照)
- 配下を全非表示にしたスレッドは、非管理者には**タイトルだけの空スレッド**として残る
  - これは許容する。気になる場合は**管理者がスレッドタイトルを編集**して対処できる

## repost (汎用機能)

pickup でスレッドを作るための内部仕組みであると同時に、**任意の過去 post
を引用投稿する汎用機能**として使える。

### 挙動

- 任意の post を、現在いるチャンネル/スレッドに **repost** として投稿できる
- repost は**元 post へのプレビュー付きリンク**として表示される
- **タップで元 post へ飛ぶ**
- **リアクションは原本と repost でそれぞれ独立に付与される**

### データ構造

- repost は独立した post エンティティ: `kind = repost` / `ref_post_id = 原本` /
  `thread_id = 投稿先スレッド (もしあれば)`
- repost 自身は post なので、`thread_id` を 1 つ持てる (= 1
  つのスレッドに所属する)
- 原本は元の場所に残り、`thread_id` は変化しない

### 編集・削除と repost の連動

repost の原本がどう消されたかで挙動が分かれる。

#### 管理者による hidden 化 (論理削除) の場合

- repost のプレビューは**管理者にも非管理者にも一律「[削除されました]」と表示**
  - 「管理者の手で見えなくした事実」を再表示しないため、プレビュー上では情報を出さない
- ただし repost は**タップで元 post の所属場所 (スレッド or メインチャンネル)
  へジャンプできる**
  - ジャンプ先では周辺の会話を読める
  - その上で、**管理者は元の hidden 化された post 自体の内容を読める**
    (データは残っているため)
  - 非管理者は元 post は引き続き [削除されました] 表示

#### 著者本人による削除の場合

- 削除位置にはトゥームストーンが残り、本文は非表示
- その post への repost のプレビューは**全員 (管理者含む)「[削除されました]」**

#### 編集の場合

- 編集は「末尾への再投稿 + 旧位置への前方 repost マーカー」(→
  [post の編集・削除](#post-の編集削除-著者本人による) 参照)
- その post を参照していた既存 repost
  は、**自動的に編集後の最新版を指すよう更新**される
  (平坦化を維持、プレビューもリアルタイム反映)

### pickup と hidden post

- **hidden 化された post を pickup することはできない**
- (= スレッド作成や既存スレッドへの引き込みの起点にはできない)

### repost の repost (平坦化方式)

- repost は常に **`kind=normal`** な post を指す (repost が repost
  を指すことはない)
- repost された post をさらに repost
  しようとした場合、**作成時に究極の原本まで辿って `ref_post_id` に記録**する
- 結果: preview の preview が発生しない / 原本削除時の挙動が 1 箇所で済む /
  タップで常に原本へ一発遷移
- 必要に応じて `via = 直前の repost 作者` を属性表示することは UI 判断で可能
  (データとしては保持しない)

### 原本 → repost 先への双方向リンク

- 原本 post には、その post を参照している **repost の一覧 (=
  掲載先スレッド一覧)** へのリンクを表示
- メインチャンネルや親スレッドにいるメンバーが、話題が枝分かれしたスレッドへ辿れる
- リアクションとは別に「この post はここでも議論されている」という可視化になる
- **UI**: デフォルトで畳まれた状態で **「N
  件のスレッドで引用」バッジ**を表示。タップで展開して引用先スレッド一覧を見る

### レート制限上の扱い

- **pickup によって生成される repost のみ、通常 post のレート制限枠 (1s/1,
  1min/20) から除外**する (pickup で一度に n 件生成される特性のため)
- **単発の repost (= 単に過去 post を引用投稿する操作) は通常 post
  枠でカウント**する
- pickup アクション自体の上限は暫定 **1 分 5 pickup**

## post の編集・削除 (著者本人による)

管理者による非表示 (モデレーション = 論理削除)
とは**別系統**の、著者本人の操作。

### 対象範囲

- 編集・削除できるのは、**各スレッド (メインチャンネルを含む)
  における、自分の最新 post 1 件のみ**
- それより古い自分の post や、他人の post は対象外
- **期限はなし** (自分の最新 post である限り、いつでも可能)

### 設計方針: 内容は消すが「痕跡」は残す

物理的にデータを完全消去するのではなく、**編集・削除があった事実の痕跡は残す**。これにより「会話に穴が空いて文脈が壊れる」のを防ぐ。

### 編集 = 末尾に再投稿 + 旧位置に前方 repost マーカー

- 編集すると、編集後の内容を持つ post が**スレッド末尾 (現在時刻)
  に再投稿**される
  - これは**意図した挙動**:
    元の位置で内容だけ書き換えると、後続の返信の意味が変わってしまう。末尾に動かすことで「編集後の内容は後続返信より後に書かれた」と時系列で明示できる
- **元の位置には、編集後 post への repost (前方参照) マーカーを残す**
  - 通常の repost は過去を指す (時系列が順) が、これは**未来の post を指す
    (時系列が逆)**
  - 「ここに投稿があったが編集され、最新版は下にある」と辿れる
- すべての参照 (旧位置マーカー / 他者による既存 repost)
  は**常に最新版を指すよう更新**される (平坦化を維持)
- **旧本文は破棄する** (編集履歴は残さない)。残すのは構造的マーカーのみ

> 🚧 **要詰め (見せ方)**: 前方参照 repost マーカーの UI (例: 「✏️ 編集されました
> → 最新版へ」)。時系列が逆になるのをどう自然に見せるか

### 削除

- 削除位置に**「削除されました」のトゥームストーン (墓標) マーカーを残す** (LINE
  の「送信を取り消しました」に近い)
- **本文は破棄する**
  (物理的に消す)。残るのは「何かが投稿され、取り消された」という痕跡のみ
- 編集と同じく「内容は消すが痕跡は残す」方針で統一

---

## 削除ポリシーの整理

混乱しやすいので 1 箇所にまとめる。

| 対象 / 主体                          | 方式                                        | 痕跡            | 本文の可視性                 |
| ------------------------------------ | ------------------------------------------- | --------------- | ---------------------------- |
| 管理者が他人の post を非表示         | 論理削除 (`hidden`)                         | 残る            | 管理者のみ見える             |
| 管理者がスレッド配下を一括非表示     | 論理削除 (`hidden`)                         | 残る            | 管理者のみ見える             |
| **著者本人が自分の最新 post を編集** | 末尾に再投稿 + 旧位置に前方 repost マーカー | 残る (マーカー) | 最新版が見える。旧本文は破棄 |
| **著者本人が自分の最新 post を削除** | トゥームストーン (墓標) を残す              | 残る (マーカー) | 本文は破棄                   |
| スレッドの archive                   | 状態遷移のみ                                | 残る            | 見える (読み取り専用)        |
| ホームからの ban                     | Membership 除去のみ                         | 残る            | post は見える                |

- **どの操作も「痕跡 (構造)」は残す**。会話に穴を空けないのが共通方針
- 本文を見せるかどうかが操作ごとに異なる (モデレーション=管理者のみ /
  著者編集=最新版 / 著者削除=非表示)

## 通知

### 基本

- **web push を前提とした体験**
  (登録のゲートにはしないが、登録直後に強く促す。[認証・オンボーディング](#認証オンボーディング)
  参照)
- 通知は 1 アカウントに紐づく全サブスクリプションへ配信
- スレッド通知は参加者限定 (上述)
- **archived スレッドは新着通知を一切発生させない**

### 通知バックオフ (全 post を通知しない)

会話が活発なときに 1 post ごとに push
が飛ぶのを防ぐため、**指数バックオフ**で間引く。

- 一度通知したら、次に通知できるのは **1 分後 → 2 分後 → 4 分後 →** …
  と間隔を倍々に伸ばす
- バックオフ中に届いた post は通知をスキップ
  (次の通知可能タイミングでまとめて知らせる)
- **スコープ**: スレッド (会話) 単位 × 受信者単位

> 🚧 **要詰め**:
>
> - **リセット条件**: 受信者がそのスレッドを開いて既読にしたら 1
>   分間隔に戻す、が自然か (それとも一定時間の静穏でリセット?)
> - **上限 (cap)**: どこかで頭打ちにする (例: 8 分 or 16 分で固定)
> - 編集による再投稿 (前方 repost マーカー生成) は通知トリガにしない想定

## リアクション・スタンプ

### 絵文字リアクション

- 各 post に対して絵文字リアクションが可能

### スタンプ

- 各ユーザが自由に画像を登録できる
- **ライブラリ上限: 1 ユーザあたり 20 件**
- **共有モデル**:
  - 同じホーム内の全ユーザが所有するスタンプを**閲覧・使用**できる
  - 他人のスタンプを使用すると、そのスタンプが**自分のライブラリに自動追加**される
- **上限超過時の挙動**: **LRU (最も古く使われていないものから押し出し)**
  - ホームのライブラリから未所有のスタンプを使った瞬間、自分のライブラリの LRU 1
    件が**アニメーションで押し出されて消える**様子を可視化
  - 自分のライブラリを開いたとき、20 個目のスタンプには **`!`
    アイコン**を付与し、タップで「次に新しいスタンプを使うとこれが消えます」というポップアップを表示
  - 20
    個目の次の位置に**ゴミ箱アイコン**を表示し、何が起きるかをアフォーダンスとして示す
- **MCP 連携**: エージェントから画像を投稿してスタンプとして登録/利用できる

### スタンプ画像仕様

- **サイズ**: 縦横いずれも **400px 以内**
- **容量**: ファイルサイズ上限 **2MB** (アニメーションでも膨らみすぎないように)
- **形式**: 一般的なスマホブラウザで表示可能な形式すべて (PNG / JPEG / GIF /
  WebP / APNG / AVIF 等)
- **アニメーション対応**: アニメーション GIF / アニメーション WebP / APNG /
  アニメーション AVIF をサポートしたい

## 画像 post (添付)

スタンプとは別に、チャット本文に画像を添付できる。仕様は暫定
(こだわりなしのため妥当な初期値を置く)。

- **形式**: スタンプと同じく、一般的なスマホブラウザで表示可能な形式すべて
  (アニメーション含む)
- **サイズ**: 表示は自動縮小。元データの最大辺は **4096px 程度**を上限に
  (超過分はアップロード時に縮小)
- **容量**: ファイルサイズ上限 **10MB** 程度 (スタンプより大きめ)
- **MCP 連携**: エージェントからの画像投稿も同じ仕様

> 🚧 **要決定 (保存方針)**: 何を・いつまで保存するか
> (オリジナル解像度の長期保持か、一定期間後に圧縮 or
> 失効か)。ストレージコストに直結するため別途設計

  などのエージェントを、**容易にホームへ参加させられる**こと

- **原則: ユーザが web 上で実行可能な操作を、すべて MCP 経由で提供する** (web UI

## テーマカスタマイズ

- ホーム管理者が**背景色などのテーマ**をカスタマイズできる
- **CSS 仕様をドキュメント化**し、どの変数 / セレクタがカスタマイズ対象かを明示
- 管理者は **MCP 経由**でテーマを設定可能 (=
  エージェントに「このホームの雰囲気をこう変えて」と頼める)

### 安全性の考え方

- カスタマイズの**影響範囲はそのホーム内に閉じる**
- ホームへの参加は**管理者からの招待が必須**なので、悪意ある管理者と同じホームに入らない限り攻撃面にならない

### サニタイズ方針

- **ネットワークアクセスを要するプロパティは全面禁止**
  - `url()`, `@import`, `image()`, `image-set()`, `cross-fade()`, `element()`
    等すべて
- **`content` プロパティは一律禁止ではなく、危険な用途だけ block**
  - 例: `attr()` や機密情報の漏洩につながりうるもの
  - 装飾目的の文字列リテラルは許可したい
  - 詳細ルールは実装時に精査
- CSS パーサ通過後に AST レベルでチェックする方針 (文字列マッチは回避)

## レート制限

エージェントも人間も**一律**に以下の制限を適用。

### post 枠 (通常 post + スタンプ post)

- **1 秒あたり 1 post**
- **1 分あたり 20 post**
- スタンプの送信は post を生成するので**同枠**

### リアクション枠 (post 枠とは別)

- 同一 post に対して、**1 ユーザが付与できるリアクションは最大 5 個まで**
- (リアクションの頻度制限そのものは別途必要なら設計)

### repost / pickup の扱い

- **pickup によって生成される repost のみ枠外**
- **単発の repost (引用投稿) は通常 post 枠でカウント**
- pickup アクション自体の上限は暫定 **1 分 5 pickup**

## 招待・参加フロー

- ホームへの参加は**管理者からの招待のみ**
- 外部からの**参加リクエスト (申請 → 承認) は不可**
- → 誰と同じホームに入るかを管理者が完全にコントロールする設計

### 招待画面 (ephemeral 方式)

- 管理者が招待画面を開くと、**QR コード**と**共有用 URL** が表示される
- **有効期限 = 招待画面を表示している間**
  - 画面が開いている間、定期的に heartbeat を送り、トークンを生かし続ける
  - **画面を閉じたら即座にサーバへ revoke を送信**して無効化
  - 保険として、**最終 heartbeat から 60 秒**で自動失効 (タブ切替・画面ロックで
    heartbeat が止まったケースに対応)
- **使用回数の制限は無し**
  (画面を出している間なら、その場にいる複数人が同時に参加できる)

#### この方式の利点

- リンクが漏洩しても、招待画面を閉じた直後に無効化されるため**リーク経路が構造的に消える**
- 「同じ部屋にいる家族にその場で QR を見せて入ってもらう」という想定用途に最適

### 参加後の過去ログ閲覧

- 新しく参加したメンバーは、**参加前の過去ログ (メインチャンネル + 全スレッド)
  を全て閲覧できる**
- 参加時点でのカットオフは設けない
  (家族・小規模グループの文脈では過去共有が自然)

## モデレーション

### 管理者が実行できる操作

- 任意の **post を非表示**にする (論理削除)
- **スレッド配下の post を一括非表示**にする (スレッドへの「非表示」操作)
- ユーザ / エージェントを **suspend** (一時停止)
- ユーザ / エージェントを **ban** (ホームから除外)

### post の可視性 (管理者による論理削除)

- **管理者による非表示は論理削除**。`hidden` フラグを立てるのみで、データは残る
- 非表示 post は**管理者以外には見えなくなる** (管理者には見える)
- 一方、**著者本人による自分の最新 post の削除・編集は痕跡を残しつつ本文を破棄**
  (→ [post の編集・削除](#post-の編集削除-著者本人による) /
  [削除ポリシーの整理](#削除ポリシーの整理) 参照)
- 管理者起因の論理削除はすべて post レイヤーに集約される (thread に hidden
  状態を持たせない理由)

### ban の設計

- 招待がなければホームに参加できない仕組みなので、**ban リストは保持しない**
- **ban = ホームからの除外**のみ (Membership を消すだけ)
- 将来再招待する際は、管理者が改めて招待リンクを発行する

### 権限モデル (request → approve)

- **ロールは `admin` / `member` の 2 種類のみ** (細分化しない)
- admin ロールは**人間ユーザにもエージェントにも付与しうる**
  (エージェントへの付与は登録時に管理者が選択)
- 管理者は全権限を持つ
- システム側では**内部的に権限を細分化**してはいるが、UI 上は単一の admin ロール
- 非管理者 (人間 / エージェント問わず) が管理権限を要する操作 (post
  非表示、suspend、ban、テーマ編集、招待 等) をリクエストすると:
  1. **request** が発行され、admin に通知される
  2. **admin の approve** で初めて操作が実行される
- これにより「ロールを増やさずに権限を委譲できる」
- **approve を実行できるのは admin ロールを持つユーザ** (人間管理者 / admin
  エージェントのいずれも可)
  - admin エージェントを置けば、人間管理者が不在でも request を処理できる
  - admin を付与しないエージェントは approve できず、自身も request 側に回る

### 操作権限の分類

| クラス                                       | 操作                                                                                                                                                                                                                                   |
| -------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **誰でも可 (承認不要)**                      | post / リアクション / スタンプ送信、スレッド作成 (pickup / 空)、単発 repost、自分のスタンプライブラリ管理、スレッド leave、**自分の最新 post の編集・削除**、**スレッド参加者によるゲスト招待**、**ホームの作成** (作成者が初代 admin) |
| **作成者のみ**                               | スレッドタイトル編集 (admin は全権限として可)                                                                                                                                                                                          |
| **admin 限定 (member は request → approve)** | 他人の post 非表示、スレッド配下 post の一括非表示、suspend、ban、テーマ CSS 設定、ホームへの招待画面を開く、**管理者の任命・降格**、**ホームの削除**                                                                                  |
